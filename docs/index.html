what it does

Four mechanisms, each with a receipt.

  • The proof gate. “Done” exists only on a fresh green exit code. Model language is optimized to be convincing, not true — self-report can't gate closure, an exit code can. It also makes cheap-model routing safe: reliability was never the model's to assert.
  • The governed mission. Headless run under a turn cap, low effort, lean env, tight output policy. −37% tokens vs raw at tied accuracy. The savings are entirely in the prompt and the flags — no heavy machinery.
  • Replay. A repeated task-class restores its cached file-state and re-proves fresh, zero model — 17.2s → 128ms. Honest limit: it only fires on repeats (CI, evals, batch); near-inert for one-off work, but idle-cost is ~nil so it stays as cheap insurance.
  • The tax meter. A read-only session beacon: per-message tokens vs the fresh floor, so you know when to start fresh (the 15–21× accumulation lever). It reports; it never refuses.

the honest limits

What it is, and isn't.

  • Not a speed chip for one-off work. First-run wall is the model's floor (turns × latency); the speed win is replay, and it fires on repeats — which is exactly where heavy automation lives.
  • A token + accuracy chip. A governor on the model — leaner fuel, verified closure — not an upgrade of it. An ECU leans the mixture; it doesn't add cylinders.
quickstart — the proof-gated lane
# author a task with a machine-checkable proof (born-green proofs are refused)
node tiny.mjs add "implement sumDigits" \
  --prove 'node -e "import(\"./solution.mjs\").then(m=>process.exit(m.sumDigits(238)===13?0:1))"' \
  --repo . --check

# drive it: claim → governed mission → FRESH proof → done  (a repeat replays in ~0.1s)
node tiny.mjs work

# or a one-off governed mission
node tiny.mjs run --model haiku -- "fix the failing test in parser.mjs"

# session context meter — when to start fresh
node tiny.mjs tax
